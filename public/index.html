BHT EMR API

General notes

The BHT EMR API is a RESTful API providing an abstraction layer to a customised OpenMRS 1.7 database and application engine for EMRs … FIXME.

Authentication

The API makes use of API keys for authentication. Almost all requests to the API must be accompanied by an API key. API keys are generated by logging into the API [see Logging in]. Keys are valid for 24 hours and a user can only have one valid key at a given point in time. Generating a new key invalidates any existing key.

Request/Response Structure

The API primarily consumes and spits out JSON. It makes use of a number of HTTP verbs for manipulating and access resources. All POST and PUT requests must be accompanied by a JSON payload and a Content-type=application/json header. Below is a summary of the verbs the API recognises.

Verb Interpretation Expected Response Status
GET Retrieve some resource.
eg. GET /api/v1/patients to retrieve patients 		200 - Ok
POST Create a resource. 201 - Created
PUT Update a resource. 200 - Ok
DELETE Destroy some resource. 204 - No Content

Successful requests get a response with a status code of either 200, 201, or 204. For 200 and 201 responses a response body is always available. The body is the object (or list of objects) that was being accessed.

Failed requests are flagged by 4XX status codes. Errors in this range generally have a response body of the following structure:

    {
        "errors": [
            "Error message",
            "Another error message",
            ...,
            "More error messages"
        ]
    }

However a 400 error for POST and PUT requests come as:

    {
        "errors": {
            "field": "This field is required"
            ...
            "some_other_field": "This field too is required"
        }
    }

Response status codes

Below is a description the response codes the API returns:

Status Code Status Text Description
200 Ok Request was successful
201 Created Resource has been created
204 No Content Resource has been deleted
400 Bad Request Received a malformed request from client
401 Unauthorized A valid API key is required
403 Forbidden Logged in user is not allowed to perform request
500 Internal Server Error API is ill, only thing that can be done is contacting the system administrator

API detail

Log in

Logging in is a way of generating an API key which must be used in subsequent requests to the API that require authorisation. The generated api key is valid for 24 hours.

NOTE: Clients are limited to one API key at a time. A log in replaces any existing api key.

Retrieve registered people

/api/v1/people

This endpoint returns a list of people registered as either patients or users. By default only 10 random people are returned. Pagination is supported through the use of the parameters page and page_size as follows:

http://<host>:<port>/api/v1/people?page=0&page_size

The example above retrieve the first 50 people.
